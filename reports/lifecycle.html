📋 README.md

v2.0 完成

Hermes Capability Pack

boku 能力模块化与跨 Agent 可复用性项目。将全部 185 项 Agent 能力拆分为标准化「能力包」。

核心概念

术语定义示例
能力包一组可移植的 Agent 能力单元文档生成包、安全审计包
技能如何完成某事的步骤化指令pdf-layout/SKILL.md
经验什么场景用什么、已知陷阱WQY 字体回退经验
MCP 配置外部工具服务的连接定义OCR 服务、翻译 API
知识库引用三层知识库的关联路径L3 Brain 概念页
适配器将能力包映射到特定 Agent 格式Hermes / Claude Code / Codex CLI

扩展性三原则

  • 按领域不按工具 — 按「解决什么问题」而非「用什么工具」
  • 模块内完整闭环 — 含怎么做 + 何时做 + 踩过什么坑
  • 扩展优先于重构 — 新能力先入槽，不完美也比丢失好

📦 SPEC-001: 模块分割方案 (v2.0)

✅ CLARIFY→RESEARCH→CREATE→QA_GATE·REVIEW 已批准

完整能力审计 — 原始遗漏分析

v1 遗漏了严重性原因
知识库系统🔴 致命只看到显式技能，忽略三层知识体系
元认知系统🔴 致命自我分析/审查是「关于能力的能力」
质量保障🔴 致命QA 门禁/测试/审查是跨领域元能力
消息平台🟡 重要飞书/微信是通信能力而非工具
音视频媒体🟡 重要TTS/图像/音乐是独立能力域
GitHub 生态🟡 重要10 个技能构成完整生态
网络代理🟢 中等Clash/代理/浏览器自动化
新闻研究🟢 中等新闻简报/Arxiv/AI 趋势
Agent 协作🟢 中等子代理/多 Agent/BMAD 编排
社交娱乐🟢 中等MC 服务器/健身/元宝
框架成长性🔴 致命没有考虑 Hermes 新版本带来新能力

扩展性设计

技能自动归属机制

每个 skill 的 tags 标签自动匹配到对应模块。
未识别的技能 → hermes-new-features 槽暂存。
满 3 技能后触发新模块创建流程。

分类决策树

新能力诞生 → ① 属于已有领域？→ 归入对应模块
           → ② 多领域交叉？→ 主模块 + 交叉引用
           → ③ 全新领域 ≥3 技能？→ 创建新模块
           → ④ 否则暂存到 hermes-new-features 槽

Capability Pack 格式示例

name: knowledge-base
version: 1.0.0
type: capability-pack
layer: meta
capabilities:
  skills:
    - id: knowledge-precipitation
    - id: knowledge-routing
    - id: hermes-knowledge-base
  experiences:
    - id: knowledge-lifecycle (type: decision-tree)
  knowledge_refs:
    - level: L3 (ref: brain/concepts/knowledge-graph.md)
growth_config:
  auto_absorb_new_skills: true
  skill_tag_mapping:
    knowledge: knowledge-base

🔄 SPEC-002: 模块生命周期管理

✅ CLARIFY→RESEARCH→CREATE→QA_GATE·REVIEW 已批准

六阶段状态机

DRAFT → ACTIVE → MATURING → STABLE → DEPRECATED → ARCHIVED
转换前置条件操作
DRAFT → ACTIVE模块清单完整、≥1 技能cap-pack review
ACTIVE → MATURING≥3 次安装反馈 / ≥1 次更新自动判断
MATURING → STABLE≥10 次安装、≤2 个已知 Issuecap-pack stabilize
STABLE → DEPRECATED有替代模块cap-pack deprecate
DEPRECATED → ARCHIVED已废弃 ≥90 天自动归档

版本号规范

MAJOR.MINOR.PATCH — 遵循语义版本

  • MAJOR: 不兼容格式变更、技能移除
  • MINOR: 新增技能、新增经验
  • PATCH: 经验修正、文档改进

预发布标签: 1.0.0-alpha.1 · 1.0.0-beta.2 · 1.0.0-rc.1

CRUD 操作

  • Create: cap-pack install ./packs/doc-engine.yaml
  • Read: cap-pack list / cap-pack show doc-engine
  • Update: cap-pack update doc-engine --to 1.2.0
  • Delete: cap-pack remove doc-engine (备份保留 7 天)

🔄 SPEC-003: 模块迭代与进化

✅ CLARIFY→RESEARCH→CREATE→QA_GATE·REVIEW 已批准

五步闭环

日常使用 → ① 触发迭代信号 → ② 评估影响范围
         → ③ 执行迭代更新 → ④ 同步通知 → ⑤ 验证回环

迭代触发条件

类型条件优先级
🔴 自动执行报错（某技能过时）高
🟡 自动效率提升（发现更快方法）中
🟡 自动知识缺口（新场景无经验）中
🔴 被动主人反馈高
🔴 被动外部 API 变更高
🟡 被动新需求中

经验的三种来源

  • 🐛 陷阱记录 (Pitfall) — 「执行步骤 X 时报错 Y，原因是 Z，解决方案是 W」
  • 🌳 决策树 (Decision Tree) — 「在场景 A 下用方案 X，场景 B 下用方案 Y」
  • 📊 对比分析 (Comparison) — 「方案 X 比方案 Y 快 3 倍，但需要额外安装 Z」

🔌 SPEC-004: 跨 Agent 适配层方案

✅ CLARIFY→RESEARCH→CREATE→QA_GATE·REVIEW 已批准

UCA 架构

Capability Pack → UCA Core (解析/验证/路由)
                ├→ Hermes Adapter → SKILL.md + config.yaml + MCP
                ├→ Claude Code Adapter → ~/.claude/skills + claude.json
                └→ Codex CLI Adapter → .codex/rules + mcp.json

适配器接口 (Python Protocol)

class AgentAdapter:
    name: str                  # "hermes", "claude-code"
    is_available: bool         # 检查目标 Agent 是否可用
    install(pack) → Result    # 安装能力包
    uninstall(name) → Result  # 卸载能力包
    update(pack, old_ver)     # 更新
    list_installed() → list   # 列出已安装

适配优先级

优先级Agent难度理由
P0Hermes⭐⭐原生环境，Skills/Profile/MCP 全支持
P1Claude Code⭐⭐⭐Skills + CLAUDE.md + MCP 三轴
P1Codex CLI⭐⭐Rules + MCP 支持
P2OpenCode / Cursor⭐⭐⭐易用性较好，工具链更全
