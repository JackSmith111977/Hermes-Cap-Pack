2 底层原理 · 四大理论支柱

2.1 微内核架构 (Microkernel Architecture)

核心思想：最小核心 + 可插拔插件。核心只提供必要的基础设施（路由、生命周期、注册），所有业务能力通过插件扩展。

// 微内核 vs 我们的架构

┌─ Microkernel Core ───────────────────────────┐
│ pre_flight Gatekeeper · 状态管理 · 依赖解析 │
└─────────────────────────────────────────────┘
│
┌────┼────┬──────────────┐
doc-engine learning-engine developer-... (18 个 Capability Pack)
│ │ │
pdf-cluster learning-cluster sdd-cluster (~70 个 Skill Cluster)
│ │ │
pdf-layout learning-workflow spec-state (~200 个 Atomic Skill)
│ │
references/ scripts/ (叶子节点)

直接验证：

📕 MoFA Framework — Microkernel + Dual-Layer Plugin
"The kernel provides only the most basic abstractions. All non-core functionality is provided through plugin mechanisms. Upper layers depend on lower layers, not the reverse."
— mofa.ai/mofa/concepts/microkernel.html
📕 Agent-Kernel — arxiv 2512.01610
"A society-centric modular microkernel architecture consisting of a foundational core and an extensible suite of application-specific plugins. This design effectively decouples the general-purpose infrastructure from application-specific logic."
— Agent-Kernel: A MicroKernel Multi-Agent System Framework, 2025

2.2 有界上下文 (Bounded Context / DDD)

核心思想：每个领域模型只在明确边界内有效。边界内术语有唯一含义，跨边界需要翻译层。Eric Evans 2003 年在《Domain-Driven Design》中提出。

映射到我们的架构：

  • 每个 Capability Pack = 一个 Bounded Context（如 doc-engine 内的「文档」有唯一含义）
  • 跨包引用 = 跨上下文通信（如 developer-workflow 引用 quality-assurance 的测试框架）
  • cap-pack.yaml = 上下文契约（声明依赖、兼容性、接口）
  • pre_flight 门禁 = 反腐败层 (Anti-Corruption Layer)
📕 DDD for AI Agents — Sebastian Sigl
"Bounded contexts are attention boundaries. They protect the model's focus. By slicing the application into vertical subdomains, each context limits the retrieval and reasoning scope. The tokens in the prompt become higher density and higher relevance."
— sebastiansigl.com/blog/bounded-contexts-cognitive-boundaries-ai-humans
📕 AI Systems Need Bounded Contexts — NILUS
"Put AI capabilities inside bounded contexts, and make model ownership explicit. Each significant domain should own the AI behavior that depends on its semantics."
— nilus.be/blog/ai_systems_need_bounded_contexts

2.3 三层本体论 (Three-Layer Ontology)

核心思想：知识组织天然分为三层——抽象分类层、语义关系层、可执行部署层。这是本体工程 (Ontology Engineering) 和信息架构 (Information Architecture) 的经典模式。

层级Colaberry AI我们的架构类比
L1 抽象层 Skill Taxonomy (9 categories) Capability Pack (18 modules) 图书馆的分类体系
L2 语义层 Skill Relation Graph (4 relationships) Skill Cluster (~70 groups) 书架上的分类标签
L3 执行层 Skill Collection (deployable units) Atomic Skill (~200 real skills) 可借阅的书籍
📕 Colaberry AI — Skill Ontology Three-Layer Design
"The top layer (Taxonomy) defines broad categorization. The middle layer (Relation Graph) instantiates specific skills and defines how they interact. The bottom layer (Collection) groups related skills into deployable units."
— colaberry.ai/aixcelerator/skills/ontology (16,896 skills live in production)
📕 OntoSkills — Neuro-Symbolic Skill Ontology
"Skills are compiled artifacts, not interpreted documents. SKILL.md files DO NOT EXIST in the agent's context. The .ttl files are self-contained, modular, pluggable ontologies."
— OntoSkills: OWL 2 + SHACL + SPARQL, ontoskills.sh

2.4 渐进式披露 (Progressive Disclosure)

核心思想：不将所有信息一次性加载到上下文，而是按需分层加载——先元数据 (100 tokens)、再指令 (5000 tokens)、最后资源 (按需)。这是 Agent Skills 开放标准的强制要求。

层级Token 预算加载时机在我们的架构中
Metadata~100 tokensAgent 启动时cap-pack.yaml 中的 skill 列表
Instructions<5000 tokensSkill 激活时SKILL.md 正文
Resources按需需要时加载references/ scripts/ templates/
📕 Agent Skills Open Standard — agentskills.io
"Compliant implementations MUST follow the three-tier loading model. At startup, scan and load ONLY YAML frontmatter. When activated, load the full instruction body. Only when needed, load scripts, references, and assets."
— agentskills.io/specification (开放标准，Anthropic/Microsoft/OpenAI 支持)
3 架构定位 · 所属模式

🏗️ 架构类型判定

我们的系统属于 微内核 + 有界上下文 + 三层本体 的混合架构：

  • 核心：pre_flight + SQS + SDD 状态机 + 依赖解析
  • 插件：18 个 Capability Pack，每个含 Skills + Experiences
  • 契约：cap-pack.yaml（声明式 manifest）
  • 生命周期：创建→质量门禁→部署→审计→退役
